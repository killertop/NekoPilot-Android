# Android 性能基线与优化报告（2026-07-23）

## 结论

本轮只保留了能够在同一模拟器、同一 Release 构建和同一测试数据上复测到收益的启动路径改动：

- 将首屏不需要的 VPN Binder 连接移到首帧提交之后。
- 将 WorkManager 初始化、应用列表枚举、孤儿节点清理和偏好持久化移到首帧提交之后。
- 首次进入首页不再在 `onViewCreated` 和第一次 `onResume` 重复读取相同状态。
- 首次节点快照直接决定空态，不再额外执行 `COUNT(*)`。
- 孤儿清理先执行无副作用的 `EXISTS`，仅在确有孤儿时执行会触发 Room 失效通知的 `DELETE`。
- 增加 `reportFullyDrawn()`，把“节点列表快照已应用且对应帧已绘制”定义为业务首屏完成。

## 测试环境与口径

- 构建：正式 Release、arm64-v8a、R8 与资源压缩开启。
- 最终包：`NekoPilot-2.2.8-arm64-v8a.apk`。
- 设备：Android 15 / API 35 arm64 模拟器 `NekoPilot_API_35`。
- 渲染：无窗口、SwiftShader；系统动画比例均为 0。
- 页面数据：首页 1 个节点，未连接 VPN。
- 冷启动：`am force-stop` 后 `am start -W`；每轮结束等待 2 秒，避免后台维护被下一轮强杀。
- 热启动：应用进程和 Activity 保留，从 Launcher 返回。
- 暖启动：Activity 任务保留，应用进程通过 `am kill` 回收后返回。
- 业务首屏：首次节点列表快照应用后，等待 RecyclerView 下一次 `onPreDraw`，并在该帧之后调用 `reportFullyDrawn()`。
- 主线程时间：atrace 中 MainActivity 的 `performCreate` slice。
- 峰值 PSS：启动后约 3 秒内，高频执行 `dumpsys meminfo --package`，合计 main、`:bg` 和 `:room` 三个进程。

模拟器使用软件 GPU，因此 `gfxinfo` 的帧耗时和卡顿率不能代表真机，本报告不据此声称 UI 绘制性能提升。

## 启动数据库与初始化证据

同一 1 秒 database atrace 中，实际执行次数如下：

| 查询 | 基线 | 优化后 |
|---|---:|---:|
| `proxy_groups` 全量查询 | 2 | 1 |
| `proxy_entities COUNT(*)` | 2 | 0 |
| 选中节点实体查询 | 3 | 2 |
| 首页节点投影查询 | 5 | 5 |

因此本轮只声称减少了分组、空态和选中节点的重复查询；首页节点投影查询次数没有下降，不声称该项改善。

trace 同时证明基线冷启动在首帧前启动了 `:bg` VpnService、`:room` Room invalidation service 和 Remote WorkManager service。优化后 VPN Binder 与维护初始化不再位于 Activity 首帧的必要路径。

## 逐项审计结果

### Compose、列表、动画和图片

项目是 Fragment + RecyclerView，不使用 Compose；重组次数、不稳定参数和 LazyColumn 均不适用。RecyclerView 已使用 stable ID、禁用批量移动动画、批量合并测速结果，并对超大列表设置 Diff 阈值。首页图片是本地矢量资源，没有图片加载库。没有数据支持继续调整。

### 主线程、文件、数据库、JSON 和网络

未发现确定发生在主线程的文件、JSON 或网络操作。Room 查询运行在 Dispatcher 线程。MainActivity `performCreate` 的中位数下降，但样本仅 5 次，最终收益以冷启动和业务首屏的 A-B-A 数据为主。

### 内存、泄漏和分配

启动峰值 PSS 没有可测改善，因此不声称内存优化。静态审计发现二维码页可能瞬时分配约 6.7 MB，PackageCache 会保存完整应用信息，但本轮没有对应页面的 heap/profile 数据，未修改。Debug 已开启 SQLite、Closeable 和 registration 泄漏检测；未发现可证实泄漏。

### Flow、轮询、定时器和后台唤醒

未发现重复 Flow 订阅、`AlarmManager`、`Timer`、`GlobalScope` 或 `observeForever`。首页流量采样为 1 Hz，仅在 STARTED 生命周期运行；未连接状态的 10 秒进程 CPU 增量为 0 tick，未优化。连接状态切换存在 100 ms、最多 60 秒的有界轮询，但没有实际达到高循环次数的证据，未修改。

### WakeLock

未连接 VPN 时 `mWakeLockSummary=0x0`。VPN 连接后代码会在整个连接期持有 Partial WakeLock；由于缺少屏灭 30–60 分钟的耗电和断流 A/B，未修改，也不声称耗电改善。

### sing-box 日志

没有日志文件轮询。自动切换开启时会消费内存中的 warn 日志流，但该功能默认关闭，本轮没有坏节点持续报错场景的 CPU 数据，未修改。

### 节点测速

默认并发 5，范围 1–10；native core 每批最多 32 个节点。静态审计发现取消协程不会主动取消阻塞的 OkHttp `execute()` 或 Socket，请求可能等待最长约 15 秒，但本轮没有可复现测速端点和取消延迟数据，未修改，也不声称取消性能改善。

### 配置解析与订阅更新

订阅下载上限 32 MiB，节点上限 10,000。解析路径可能同时保留原文、Base64 结果、节点实体和序列化 identity，但没有 100/1,000/10,000 节点的峰值 heap 与阶段耗时数据，未进行无依据重写。

### APK 与原生库体积

基线 APK 中 `libbox.so` 解压 58,078,448 B、压缩 18,571,662 B，占 APK 文件约 85.4%。DEX 压缩 1,907,701 B，resources 压缩 1,013,346 B，assets 压缩 92,966 B。

项目已启用 R8、资源压缩、arm64 单 ABI和语言过滤；libbox 已使用 `-s -w -trimpath` 且无符号表。当前没有证据支持继续 strip 或删除 gVisor、QUIC、WireGuard、uTLS、Clash API 等构建 tag。本轮不同 Git revision 元数据的优化构建比基线增加约 1.1 KB，因此明确不声称包体积改善。

## 尚未量化、不得声称已改善

- 真机启动、帧稳定性、GPU 渲染和滚动性能。
- 100/1,000/5,000 节点的首屏、排序、分配和滚动数据。
- 10,000 节点订阅更新的阶段耗时、GC 和峰值内存。
- VPN 屏灭耗电、深度休眠、断流率和 WakeLock 调整。
- 自动切换高频 warn 日志的 `:bg` CPU。
- 测速取消延迟、残留 Socket、临时 core 和内存回收时间。
- 二维码页和应用代理页的峰值分配与退出后回收。

以上项目在取得同设备、同数据、同场景的前后数据前，不应标记为“性能已提升”。
